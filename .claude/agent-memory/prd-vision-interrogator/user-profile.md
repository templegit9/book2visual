The user is the sole owner and developer of Book2Visual, a personal/indie macOS project. They are technically capable (comfortable with SwiftUI, Python/FastAPI, SSH, GPU infra) but building this as a side project with a frugal mindset — cost consciousness is a core constraint, not a nice-to-have.

**Role:** Solo indie developer / product owner.

**Domain knowledge:** Familiar with LLM/image-model concepts (knows Qwen, FLUX, IP-Adapter by name), GPU rental platforms (ThunderCompute), and macOS native development (SwiftUI, Keychain, SSH tunnels).

**Working style preferences:** Willing to do one-time manual setup steps (e.g. model weight baking) if it reduces per-run cost and complexity. Values pragmatic scope cuts — accepted a clean v1 scope boundary early. Open to being directed on opinionated architecture choices when they have no strong preference.

**Budget mindset:** "$0-ish" — satisfied by cheap GPU rental (ThunderCompute pay-per-use), all models open-source/local to the instance. Avoiding cloud API costs per run is a design goal.

**Privacy stance:** Strong preference for local/private processing — no book text sent to third-party APIs. Output is for personal use only.

**v1 ambition:** Demo-grade (2,000–4,000 word short story input), not production-scale. Primary goal is to validate the pipeline and personal enjoyment, not commercial launch.
